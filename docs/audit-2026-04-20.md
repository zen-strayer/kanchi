## What It Does

Kanchi is a real-time Celery task monitoring tool. The backend connects to a Celery broker (RabbitMQ or Redis) using `celery.events.Receiver`, captures all standard task lifecycle events, persists them to SQLite/PostgreSQL/MySQL, and broadcasts over WebSocket. The Nuxt.js frontend displays a live dashboard, task registry, worker status panel, and a workflow automation builder.

**Stack:** Python 3.12 / FastAPI / SQLAlchemy 2 / Alembic / Celery 5 + Nuxt 4 / Pinia / Tailwind

---

## What It Does Well

### Core event capture
The Celery event capture using `Receiver.capture()` is correctly implemented. Handles all standard task lifecycle events (`sent`, `received`, `started`, `succeeded`, `failed`, `retried`, `revoked`) plus worker heartbeats and two custom progress event types (`kanchi-task-progress`, `kanchi-task-steps`). The event-to-WebSocket pipeline correctly crosses the sync/async boundary via `loop.call_soon_threadsafe` + `asyncio.Queue` without blocking the event loop.

### Database design
The `task_latest` snapshot table is a smart denormalization — one row per task with the most recent state — enabling fast aggregated list queries without window functions. Well-indexed (10 composite indexes). Dialect-specific upsert (PostgreSQL / MySQL / SQLite) is handled correctly in `task_service.py`.

### Workflow automation
The trigger→condition→action engine is functional: evaluates Celery events against user-defined conditions (12 operators), runs Slack notifications and task retries, with circuit breaker protection to prevent retry storms. `root_id` is preserved across retries so the circuit breaker correctly groups the entire chain.

### Retry tracking & orphan detection
Retry relationship chain model is correct. Orphan detection uses two complementary mechanisms — event-driven (worker-offline events) and periodic health check (15s poll, 30s timeout) — which provides good coverage.

### No data exfiltration
Every outbound HTTP call is either user-triggered OAuth (Google/GitHub) or user-configured Slack webhooks. **Zero analytics SDKs, zero telemetry, zero phone-home behavior.** Confirmed clean after reviewing all Python and TypeScript source files.

### Auth (when enabled)
Custom HMAC-SHA256 tokens with DB-stored hash for revocation support, Google + GitHub OAuth with popup/postMessage flow, `allowed_email_patterns` gating. Structurally correct.

### Migrations
23 Alembic migration files with descriptive names covering the full schema evolution. Auto-runs on startup.

## Where It Falls Apart (Bugs)

These are confirmed bugs in the current codebase, not missing features.

### 1. Monitor thread has no reconnect logic — `agent/monitor.py:169`
If the broker drops, `recv.capture()` raises, the daemon thread exits silently, and monitoring stops permanently. The app continues serving stale data while reporting itself healthy. **Process restart required to recover.**

### 2. WebSocket broadcaster reconnect bug — `agent/connection_manager.py:70`
`_running` flag is never reset after the first client disconnects. `start_background_broadcaster()` no-ops on subsequent connections. Events pile up in an unread queue after reconnect.

### 3. Orphan detection doesn't update `task_latest` — `agent/services/orphan_detection_service.py:62`
Bulk UPDATE sets `is_orphan=True` on `task_events` rows, but the `task_latest` snapshot (which drives the aggregated task list view) is never updated. Orphaned tasks show wrong state in the UI until they receive a new event.

### 4. `avg_runtime` math is wrong — `agent/services/daily_stats_service.py:117`
Incremental average formula double-counts historical values. Daily average runtime stats are quietly incorrect.

### 5. SQLite-only datetime expression — `agent/services/task_service.py:395`
`func.datetime('now', '-1 hour')` is SQLite syntax and crashes silently on PostgreSQL/MySQL. The "recent activity" counter on the dashboard breaks when switching off SQLite.

### 6. Frontend WebSocket gives up permanently after 5 reconnect failures — `frontend/app/stores/websocket.ts:34`
After 5 failed reconnects, `canReconnect` is permanently `false` with no recovery path except a page reload. No user notification is shown.

### 7. `time.sleep(2)` while holding a DB session — `agent/event_handler.py:119`
The orphan grace-period sleep runs with an open SQLAlchemy session. On SQLite (the default), this serializes all writes for 2 seconds every time a worker goes offline.

## Where It Does Things Poorly (Design Weaknesses)

### Workflow engine spawns unbounded threads — `agent/services/workflow_engine.py:45`
Each Celery event spawns a new `threading.Thread` + `asyncio.run()`. A busy Celery deployment (hundreds of events/second) will exhaust OS thread limits. No pool, no queue depth limit, no backpressure.

### No data retention/TTL mechanism
`task_events`, `workflow_executions`, and `task_progress_events` grow forever. No archival, pruning, or partition scheme. Long-running instances will bloat indefinitely.

### `CELERY_BROKER_URL` not validated at startup — `agent/config.py:57`
`os.getenv('CELERY_BROKER_URL')` returns `None` if unset, silently falling back to Celery's default `amqp://guest@localhost//` rather than failing fast with a clear error.

### Environment filtering is entirely client-side — `agent/api/websocket_routes.py:144`
The code even has a comment acknowledging it: all events stream to every WebSocket client regardless of active environment, with the frontend filtering locally. Becomes a bandwidth problem at scale.

### `retried_by` stored as JSON-serialized text
Stored as `Text` in `TaskEventDB` (`agent/database.py:77`) but as native `JSON` in `TaskLatestDB`. Inconsistency requires manual `json.loads()` on every read.

### Pydantic v1/v2 mixed
`workflow.dict()` and `model_dump()` coexist throughout workflow code. Pydantic v2's `dict()` is deprecated.

### Zero test coverage on the critical path
149 tests exist, but they cover only isolated service-layer units. **Zero coverage for:** `ConnectionManager`, `EventHandler`, `CeleryEventMonitor`, any API route handler, auth flows, `WorkflowEngine` integration, `MetricsCollector`.

### `task-rejected` events not handled
Tasks rejected at the broker (serialization failure, policy) silently disappear from monitoring — they appear in-flight forever.

### `kanchi-sdk` is a black box
First-party `0.0.1` PyPI package by the upstream author with no changelog, no upgrade history, and no source in this repo. Used for optional progress reporting. Cannot be audited or controlled.

### Other code quality issues
- `from kanchi_sdk import send_kanchi_progress` (`agent/app.py:25`) — imported but never called anywhere
- `StaticPool` imported in `agent/database.py:22` but never used
- `asyncio.get_event_loop()` deprecated in `agent/connection_manager.py:25` — should be `asyncio.get_running_loop()`
- Bare `except:` clauses in `agent/models.py:94,112` swallow `KeyboardInterrupt`/`SystemExit`
- f-string and `%s` logging styles mixed throughout
- `declarative_base()` from `sqlalchemy.ext.declarative` (`agent/database.py:20`) is deprecated since SQLAlchemy 1.4

## Security & Data Safety

### No data leakage confirmed ✅
All outbound HTTP is OAuth (user-triggered) or user-configured Slack webhooks. No analytics, no tracking, no third-party SDKs.

### Security findings by severity

| Severity | Finding | Location |
|----------|---------|----------|
| HIGH | `ENABLE_PICKLE_SERIALIZATION=true` is an RCE vector — pickle deserialization of broker messages can execute arbitrary code | `agent/config.py:93`, `agent/monitor.py:34` |
| HIGH | Docker container runs as root — no `USER` instruction | `Dockerfile` |
| HIGH | Auth disabled by default (`AUTH_ENABLED=false`) — full API and WebSocket open to anyone | `agent/config.py:84` |
| HIGH | Dockerfile hardcodes `amqp://guest:guest` as default broker URL | `Dockerfile:43` |
| MEDIUM | No brute-force / rate-limiting on login endpoint | `agent/api/auth_routes.py:83` |
| MEDIUM | Empty `ALLOWED_EMAIL_PATTERNS` permits any OAuth user to log in | `agent/security/auth.py:233` |
| MEDIUM | JWT bearer token exposed in WebSocket URL query param (appears in logs/proxies) | `agent/api/websocket_routes.py:67` |
| MEDIUM | No `limit` ceiling on WebSocket `get_stored` message — potential DoS | `agent/api/websocket_routes.py:169` |
| MEDIUM | Slack webhook URL not restricted to Slack domains — SSRF-adjacent | `agent/services/action_config_service.py:123` |
| MEDIUM | Starlette 0.27.0 has CVE-2024-47874 (DoS via multipart, affects < 0.40.0) | `pyproject.toml` |
| LOW | Plaintext `BASIC_AUTH_PASSWORD` accepted in env var | `agent/config.py:97` |
| LOW | No upper bound on `limit` parameter in REST API endpoints | `agent/api/task_routes.py:69` |
| LOW | `sort_by` not whitelisted — column name enumeration possible | `agent/services/task_service.py:1244` |
| LOW | JWT tokens stored in `localStorage` — XSS-vulnerable | `frontend/app/stores/auth.ts:54,60` |
| LOW | `/metrics` Prometheus endpoint has no auth regardless of `AUTH_ENABLED` | `agent/api/metrics_routes.py:11` |
| INFO | `SESSION_SECRET_KEY` ephemeral randomization invalidates sessions on restart | `agent/config.py:131` |
| INFO | Wildcard CORS + `allow_credentials=true` (mitigated by browser spec) | `agent/app.py:78` |

## Priority Fix List

These are the fixes needed before production use, in priority order:

1. **Add broker reconnect loop in `agent/monitor.py`** — exponential backoff wrapping `recv.capture()`
2. **Fix `task_latest` update in orphan detection** — `agent/services/orphan_detection_service.py:62`
3. **Fix broadcaster `_running` flag reset** — `agent/connection_manager.py:70`
4. **Add `USER` instruction in Dockerfile** — run as non-root
5. **Set `SESSION_SECRET_KEY` and `TOKEN_SECRET_KEY` explicitly** — don't rely on the random fallback; will invalidate sessions on restart
6. **Enable `AUTH_ENABLED=true` in deployment** — default is wide open
7. **Fix `func.datetime('now', '-1 hour')`** — use dialect-agnostic expression for PostgreSQL compatibility (`agent/services/task_service.py:395`)
8. **Upgrade starlette** — current version has a known DoS CVE (pull via FastAPI upgrade)
9. **Add WebSocket reconnect reset** — allow recovery after max attempts (e.g., on `visibilitychange`)
10. **Add data retention/pruning** — `task_events` grows unbounded

## Verdict

**Fork it, fix the critical gaps, and host it privately.**

The MIT license gives full freedom to fork, modify, and host privately without restriction. The tool does its core job well and has a genuinely clean data safety story — no analytics, no tracking, no third-party calls outside what the user explicitly configures.

The bugs are real but fixable. They stem from a solo developer moving fast, not from fundamental architectural problems. The monitoring surface area (real-time task visibility, retry chains, orphan detection, workflow automation) is genuinely hard to find in alternatives — Flower is the only mature option and it covers less ground.

**Finding a replacement would likely cost more than fixing this.**

The one thing to stay ahead of: this project is effectively dormant. Upstream has ~1 commit in the last 6 months and no community momentum. If you fork privately, plan to own it long-term.
